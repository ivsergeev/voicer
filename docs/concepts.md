# Voicer & OpenVoicer — Концепции и руководство

## Что это такое

Voicer и OpenVoicer — это пара десктопных приложений, которые вместе превращают голос в инструмент управления AI-кодинг-агентом. Вы говорите в микрофон — агент пишет код.

**Voicer** занимается распознаванием речи. Он работает полностью офлайн: аудио не покидает ваш компьютер. Вы удерживаете горячую клавишу, диктуете команду, отпускаете — и через долю секунды получаете текст. Этот текст можно либо вставить прямо в позицию курсора (как диктовщик), либо отправить подключённому клиенту по WebSocket.

**OpenVoicer** — это мост между Voicer и OpenCode (AI-кодинг-агент). Он принимает распознанный текст от Voicer, при необходимости собирает контекст из нескольких сообщений, и отправляет всё это агенту как промпт. Ответ агента отображается во всплывающем уведомлении с поддержкой markdown-форматирования.

```
┌─────────┐   WebSocket    ┌─────────────┐   HTTP/SSE    ┌──────────┐
│  Voicer │ ───────────▶ │  OpenVoicer │ ──────────▶ │ OpenCode │
│ (STT)   │   claim+push   │  (роутер)   │  prompt/event │  (agent) │
└─────────┘                └─────────────┘              └──────────┘
     ▲                                                        │
     │  горячие клавиши                          ответ агента  │
     │  + микрофон                               (SSE stream)  │
  ┌──┴──┐                                                     ▼
  │ User│ ◀──────────── попап-уведомления ◀───────────────────┘
  └─────┘
```

Важно понимать, что каждое приложение самодостаточно. Voicer прекрасно работает как автономный диктовщик — вы можете диктовать текст в любое приложение на компьютере, не устанавливая OpenVoicer вообще. А OpenVoicer, в свою очередь, не привязан жёстко к Voicer — любой WebSocket-клиент, который реализует простой протокол сообщений, может выступать источником текста.

## Ключевые концепции

### 1. Push-to-talk: нажми и говори

В отличие от приложений с постоянно включённым микрофоном, Voicer использует модель push-to-talk. Вы физически контролируете, когда микрофон слушает: удерживаете клавишу — идёт запись, отпускаете — начинается распознавание. Никакого VAD (voice activity detection), никаких ложных срабатываний на фоновый шум или разговоры коллег.

На практике это работает так. Вы формулируете мысль, нажимаете горячую клавишу, произносите фразу, отпускаете. Весь цикл занимает обычно 2-5 секунд. Приложение проходит через три состояния:

```
Idle ──[нажатие]──▶ Recording ──[отпускание]──▶ Processing ──[готово]──▶ Idle
```

В состоянии **Idle** микрофон выключен, приложение ждёт нажатия горячей клавиши. В **Recording** идёт захват аудио — иконка в трее меняет цвет, чтобы вы видели, что запись активна. В **Processing** запускается нейросетевое распознавание — это занимает от 100мс до пары секунд в зависимости от длины фразы и мощности процессора. После распознавания приложение возвращается в Idle, а результат либо вставляется в курсор, либо отправляется по WebSocket.

Если вы случайно нажали клавишу и не хотите отправлять пустую запись — не беда. Voicer автоматически отбрасывает пустые распознавания в режиме вставки. В WS-режиме пустое сообщение тоже может быть полезным, если к нему прикреплён тег или контекст.

### 2. Два режима работы: вставка и отправка

Voicer предлагает два принципиально разных способа использования распознанного текста.

**Режим вставки (Insert)** — это классическая диктовка. Вы удерживаете F6, говорите, отпускаете — и текст появляется там, где стоит курсор. Технически Voicer копирует распознанный текст в буфер обмена, эмулирует Ctrl+V (или Cmd+V на macOS), а затем восстанавливает предыдущее содержимое буфера. Это работает в любом приложении: текстовом редакторе, браузере, мессенджере, терминале — везде, где можно вставить текст из буфера.

Режим вставки полностью автономен. Ему не нужен ни OpenVoicer, ни подключённые WS-клиенты, ни интернет. Просто запустите Voicer — и диктуйте.

**Режим WebSocket (WS)** — это отправка распознанного текста подключённым клиентам. Вместо вставки в курсор текст уходит по сети. Этот режим используется для взаимодействия с AI-агентом: вы диктуете команду, OpenVoicer её получает и пересылает в OpenCode.

В отличие от режима вставки, WS-режим поддерживает несколько горячих клавиш с разным поведением. F7 может просто отправлять голосовую команду, F8 — захватывать выделенный текст вместе с голосом, F9 — отправлять только тег без записи. Подробнее об этом — в следующей секции.

### 3. Расширяемые WS-хоткеи

Одна из самых мощных возможностей Voicer — произвольное количество настраиваемых WS-горячих клавиш. Каждая клавиша может быть настроена на один из трёх типов действий:

**TranscribeAndSend** — базовый режим «надиктуй и отправь». Вы удерживаете клавишу, говорите, отпускаете — распознанный текст отправляется клиенту. В качестве контекста автоматически прикладывается текущее содержимое буфера обмена. Это удобно, когда вы скопировали что-то перед тем, как дать команду — агент увидит и вашу голосовую команду, и скопированный текст.

**TranscribeWithContext** — «надиктуй с выделенным текстом». Этот режим делает кое-что хитрое: в момент нажатия горячей клавиши Voicer параллельно с записью пытается захватить выделенный текст в активном окне. Он сохраняет текущий буфер обмена, очищает его, эмулирует Ctrl+C и в течение 500 миллисекунд опрашивает буфер, ожидая появления нового текста. Если текст появился — это и есть выделенный фрагмент. Если нет — Voicer использует то, что было в буфере до этого. После захвата оригинальное содержимое буфера восстанавливается.

Результат: агент получает и вашу голосовую команду, и выделенный код. Типичный сценарий — выделить функцию в редакторе, нажать F8 и сказать «объясни что делает этот код» или «добавь обработку ошибок».

**SendTag** — «отправь тег без записи». Никакой записи не происходит. При нажатии клавиши Voicer мгновенно отправляет сообщение с заданным тегом и текущим буфером обмена в качестве контекста. Это полезно для быстрых действий: создание новой сессии, добавление скопированного кода в контекст, подтверждение действия.

Каждому хоткею можно назначить произвольный **тег** — строку, которая будет добавлена к сообщению. Voicer теги не интерпретирует, он просто транслирует их клиенту. Семантику определяет принимающая сторона. Например, OpenVoicer распознаёт теги `new-session` и `context` и обрабатывает их по-особому.

Вот пример реальной раскладки клавиш:

- **F7** → TranscribeAndSend, без тега — базовая голосовая команда агенту
- **F8** → TranscribeWithContext, без тега — команда с выделенным кодом
- **F9** → SendTag, тег `context` — быстро добавить буфер в контекст
- **F10** → TranscribeWithContext, тег `context` — надиктовать контекст с захватом выделения
- **F11** → SendTag, тег `new-session` — создать новую сессию одной кнопкой

Такая раскладка позволяет полностью управлять агентом голосом, не переключаясь в его окно.

### 4. Claim-протокол: один активный клиент

К WebSocket-серверу Voicer может подключаться произвольное количество клиентов — но транскрипции получает только один из них, тот, кто в данный момент является **активным** (claimed). Это ключевое архитектурное решение, которое предотвращает дублирование команд, когда к Voicer подключены несколько инструментов одновременно.

Протокол работает по принципу «последний claim выигрывает». Любой клиент может отправить сообщение `{"type": "claim"}`, и он станет активным. Если до этого активным был другой клиент — тот получит уведомление `{"type": "claimed", "active": false}`. Это позволяет клиентам корректно реагировать на потерю активности.

Важная деталь: при отключении активного клиента claim **не передаётся** автоматически следующему. Это сделано намеренно. Автоматическая передача могла бы привести к тому, что команды неожиданно начнут приходить клиенту, который к этому не готов. Вместо этого claim просто сбрасывается, и любой желающий клиент может явно запросить его.

Статусы и ошибки, в отличие от транскрипций, транслируются **всем** подключённым клиентам — чтобы каждый мог отображать состояние микрофона.

OpenVoicer автоматически отправляет `claim` сразу после подключения к Voicer. Для пользователя это прозрачно — достаточно запустить оба приложения, и они сами найдут друг друга.

### 5. Контекст: передача дополнительной информации

Голосовая команда сама по себе часто недостаточна. Когда вы говорите «исправь эту функцию», агенту нужно знать, какую именно функцию вы имеете в виду. Для этого существует механизм контекста — дополнительных данных, которые передаются вместе с голосовой командой.

Voicer поддерживает два источника контекста:

**Буфер обмена** — то, что вы скопировали перед тем, как дать команду. В режиме TranscribeAndSend буфер автоматически прикладывается к сообщению. Типичный сценарий: скопировали сообщение об ошибке из терминала, нажали F7, сказали «почему это происходит».

**Выделенный текст** — текст, выделенный в активном окне в момент нажатия горячей клавиши. Доступен только в режиме TranscribeWithContext. Voicer автоматически копирует выделение через эмуляцию Ctrl+C, при этом оригинальный буфер обмена сохраняется и восстанавливается после операции. Приоритет: если удалось захватить выделение — используется оно, если нет — берётся текущий буфер.

Контекст передаётся в отдельном поле `context` JSON-сообщения. Voicer не объединяет контекст с текстом — это ответственность принимающего клиента. OpenVoicer конкатенирует контекст и текст через двойной перевод строки при формировании промпта.

### 6. Теги: семантическая разметка сообщений

Тег — это произвольная строка, которую можно назначить WS-горячей клавише. Она добавляется к каждому сообщению, отправленному этой клавишей, в поле `tag` JSON-объекта. Voicer не придаёт тегам никакого значения — для него это просто строка. Всю интерпретацию делает клиент.

Это даёт огромную гибкость. Вы можете создать собственные теги для любых сценариев, и ваш клиент (или OpenVoicer) будет обрабатывать их так, как вам нужно. OpenVoicer из коробки понимает два тега:

**`new-session`** — при получении сообщения с этим тегом OpenVoicer создаёт новую сессию OpenCode. Старая сессия при этом удаляется, чтобы не плодить заброшенные сессии. Если к сообщению прикреплён текст — он отправляется как первая команда в новой сессии.

**`context`** — сообщение с этим тегом не отправляется агенту сразу. Оно сохраняется в локальной коллекции и отображается как попап-карточка. Когда приходит следующее сообщение без тега `context`, все накопленные контексты объединяются с ним и отправляются одним промптом. Подробнее — в следующей секции.

Оба тега настраиваются в настройках OpenVoicer, так что вы можете использовать любые имена.

### 7. Накопление контекста: сбор информации перед командой

Иногда одного выделенного фрагмента недостаточно. Представьте ситуацию: вам нужно показать агенту код из трёх разных файлов и попросить его что-то с ними сделать. В обычном текстовом интерфейсе вы бы копировали и вставляли фрагменты один за другим. С голосовым управлением это было бы неудобно — пришлось бы надиктовывать всё одной длинной фразой.

Накопление контекста решает эту проблему. Вы отправляете несколько сообщений с тегом `context`, и OpenVoicer собирает их в очередь, не отправляя агенту. Каждое контекстное сообщение отображается как попап с заголовком «Контекст» — вы видите, что именно накопилось. Если какой-то фрагмент добавлен по ошибке, его можно удалить прямо из попапа кнопкой «Удалить».

Когда вы готовы дать команду — просто отправляете обычное сообщение (без тега `context`). OpenVoicer берёт все накопленные контексты, добавляет к ним текст вашей команды и отправляет всё одним промптом. Контекстные попапы автоматически закрываются, коллекция очищается.

Вот типичный рабочий цикл:

```
Шаг 1: Выделили код в файле А, нажали F9
        → попап «Контекст: <код из файла А>»

Шаг 2: Открыли файл Б, выделили другой фрагмент, нажали F10, сказали «и вот этот класс»
        → попап «Контекст: и вот этот класс + <код из файла Б>»

Шаг 3: Нажали F7, сказали «зарефактори оба класса, выдели общую логику»
        → все контексты + команда отправляются одним промптом
        → контекстные попапы закрываются
        → появляется попап «Обработка...»
```

Агент получает один связный промпт: код из файла А, код из файла Б с пояснением «и вот этот класс», и финальную команду «зарефактори оба класса, выдели общую логику». Всё в правильном порядке, ничего не потерялось.

### 8. Сессии OpenCode: управление диалогами

OpenCode работает с понятием сессий — каждая сессия представляет собой отдельный диалог с агентом. OpenVoicer берёт на себя управление жизненным циклом сессий, чтобы вам не приходилось думать об этом.

При первом промпте OpenVoicer автоматически находит существующую сессию или создаёт новую. ID сессии кэшируется — последующие промпты отправляются в ту же сессию без лишних обращений к API. Если сессия была удалена извне (например, вы закрыли её в TUI OpenCode), OpenVoicer обнаружит это по ответу 404 и при следующем промпте создаст новую сессию.

Когда вам нужно начать с чистого листа — отправьте сообщение с тегом `new-session`. OpenVoicer создаст новую сессию, а старую автоматически удалит, чтобы не копились заброшенные диалоги. Если к сообщению прикреплён текст, он станет первой командой в новой сессии. Это удобно для переключения между задачами: «новая сессия, давай займёмся авторизацией».

### 9. Отмена обработки: прерывание долгих запросов

Иногда агент начинает генерировать длинный ответ, а вы уже поняли, что вопрос был задан неправильно. Или агент просто «зависает» на сложной задаче. Для таких случаев OpenVoicer предоставляет механизм отмены прямо из попап-уведомления.

Пока агент работает, вы видите попап «Обработка...» с кнопкой «Отменить». При нажатии кнопки происходит следующее:

1. OpenVoicer отправляет HTTP-запрос `POST /session/{id}/abort` для прерывания текущей генерации
2. Попап переходит в состояние «Отмена...» — вы видите, что запрос на отмену отправлен
3. OpenVoicer ожидает подтверждения отмены через SSE-событие `session.idle`
4. Когда подтверждение приходит — появляется попап «Отменено» без текста ответа агента
5. Если подтверждение не приходит в течение 10 секунд — OpenVoicer принудительно закрывает попап обработки и показывает «Отменено»

Таймаут в 10 секунд — страховка на случай, если OpenCode по какой-то причине не ответил на запрос отмены. Без него пользователь мог бы остаться с зависшим попапом «Отмена...» навсегда.

### 10. Управление процессом OpenCode

OpenVoicer умеет запускать OpenCode как дочерний процесс — вам не нужно вручную открывать терминал и набирать команды. Поддерживаются два режима запуска.

**Прямой запуск (Windows)** — OpenVoicer выполняет `opencode serve --port {port}` напрямую. Для этого нужно, чтобы `opencode` был доступен в PATH. Рабочий каталог устанавливается в указанную в настройках папку.

**Запуск через WSL2** — для тех, кто использует Linux-окружение для разработки. Этот режим сложнее технически. OpenVoicer запускает `wsl bash -i` и через stdin отправляет команды: переход в рабочую директорию и запуск opencode. Используется interactive bash (флаг `-i`), потому что non-interactive bash пропускает загрузку `.bashrc` — а именно там обычно настраивается PATH, nvm, go и другие инструменты разработки. Без interactive bash OpenCode просто не найдётся в PATH.

Команда `exec opencode serve` заменяет процесс bash на opencode. Это важно для корректного завершения: когда OpenVoicer останавливает процесс через `kill`, сигнал приходит непосредственно opencode, а не оболочке.

Перед запуском OpenVoicer проверяет, не занят ли порт. Если на порту уже что-то слушает — предполагается, что OpenCode запущен вручную. OpenVoicer не пытается запустить второй экземпляр и просто подключается к существующему.

Из трей-меню можно также открыть **TUI** (терминальный интерфейс) OpenCode — это полноценный текстовый интерфейс агента в отдельном окне терминала. OpenVoicer запускает `opencode attach` в Windows Terminal (или в cmd.exe как fallback). TUI полезен, когда вам нужно посмотреть подробную историю диалога, отредактировать настройки агента или выполнить действия, которые неудобно делать голосом.

### 11. Попап-уведомления: обратная связь без переключения окон

Оба приложения общаются с пользователем через всплывающие уведомления в правом нижнем углу экрана. Попапы намеренно неинтрузивны — они не перехватывают фокус и не мешают работе. Несколько попапов могут отображаться одновременно, выстраиваясь в стопку.

OpenVoicer использует шесть типов попапов:

**«Обработка...»** — появляется, когда промпт отправлен агенту. Содержит превью отправленного текста и кнопку «Отменить». Попап остаётся на экране, пока агент работает — это единственный визуальный индикатор того, что запрос в процессе. При отправке нескольких сообщений подряд каждое получает свой попап обработки.

**«Готово»** — появляется, когда агент завершил работу. Содержит текст ответа агента. Клик по попапу раскрывает его до полного размера с markdown-форматированием: заголовки, блоки кода с подсветкой, жирный текст. В раскрытом состоянии текст можно выделять и копировать. Одновременно отображается только один попап с ответом — при появлении нового ответа предыдущий автоматически закрывается. Этот попап не закрывается по таймеру — он остаётся, пока пользователь не нажмёт «OK» или пока не придёт новый ответ.

**«Контекст»** — отображается для каждого сообщения, добавленного в контекстную коллекцию. Содержит текст фрагмента и кнопку «Удалить». Попап живёт, пока контекст не будет отправлен (вместе со следующей командой) или удалён вручную.

**«Отменено»** — краткое уведомление об успешной отмене запроса. Закрывается автоматически по таймеру.

**«Новая сессия»** — подтверждение создания новой сессии с ID сессии. Закрывается автоматически.

**«Ошибка»** — сообщение об ошибке (проблемы с подключением, отправкой и т.п.). Требует ручного закрытия.

### 12. Иконка в системном трее: состояние с первого взгляда

Оба приложения живут в системном трее и генерируют иконки динамически через SkiaSharp. Цвет иконки мгновенно показывает текущее состояние — не нужно открывать приложение, чтобы понять, что происходит.

Иконка **Voicer** меняет цвет в зависимости от состояния микрофона. Серый означает ожидание. Голубой — есть подключённый активный WS-клиент, система готова к работе. Зелёный — идёт запись в режиме вставки. Синий — запись в WS-режиме. Оранжевый — распознавание в процессе. Красный — модель не загружена, приложение не может работать.

Иконка **OpenVoicer** имеет индикатор-точку, отражающую состояние подключений. Серая точка — не подключено ни к Voicer, ни к OpenCode. Жёлтая — подключено что-то одно (например, Voicer подключен, но OpenCode ещё не запущен). Зелёная — оба подключения активны, система полностью функциональна.

### 13. Офлайн распознавание: приватность и скорость

Voicer использует нейросетевую модель GigaAM v3 для распознавания русской речи. Модель работает через библиотеку sherpa-onnx и выполняется полностью локально — аудио не покидает компьютер, не отправляется ни в какое облако, не требует интернета.

Модель поставляется в формате ONNX с int8-квантизацией, что обеспечивает хороший баланс между качеством распознавания и скоростью работы. При запуске Voicer загружает модель в память и выполняет «прогрев» (warmup) — один цикл распознавания на тишине, чтобы первое реальное распознавание не было медленнее последующих.

Аудио записывается на частоте 16 кГц в формате моно PCM. Перед подачей в модель к аудио опционально применяется пиковая нормализация — это помогает при работе с тихим или слишком громким микрофоном. Количество CPU-потоков для инференса настраивается (по умолчанию 4); на мощных машинах можно увеличить для более быстрого распознавания длинных фраз.

GigaAM v3 поддерживает русский язык с автоматической расстановкой пунктуации. Это значит, что распознанный текст приходит с точками, запятыми и вопросительными знаками — его не нужно дополнительно обрабатывать перед отправкой агенту.

## Примеры использования

### Базовый сценарий: диктовка текста

Самый простой способ использования Voicer — как диктовщик. Запустите приложение, откройте любой текстовый редактор, поставьте курсор в нужное место. Удерживайте F6, говорите — текст появится в позиции курсора, как если бы вы его набрали. Voicer восстанавливает буфер обмена после вставки, так что ваши скопированные данные не пострадают.

Этот режим хорош для написания комментариев в коде, сообщений в чатах, заметок — всего, где набор текста занимает время. Распознавание русской речи работает достаточно хорошо для повседневного использования, хотя техническая терминология иногда требует коррекции.

### Голосовое управление AI-агентом

Запустите Voicer и OpenVoicer. OpenVoicer автоматически подключится к Voicer по WebSocket и запустит OpenCode (если включён автозапуск). Когда иконка OpenVoicer покажет зелёную точку — система готова.

Удерживайте F7, скажите: «создай функцию, которая принимает список чисел и возвращает только чётные». Отпустите. Появится попап «Обработка...» — агент работает. Через несколько секунд появится «Готово» с ответом. Кликните по попапу, чтобы увидеть полный ответ с отформатированным кодом.

### Команда с контекстом: «объясни этот код»

Выделите фрагмент кода в редакторе. Удерживайте F8, скажите «объясни, что делает этот код». Отпустите. Voicer захватит выделенный текст параллельно с записью голоса. Агент получит и код, и вашу просьбу объяснить его.

Это работает с любым текстом в любом приложении — код из IDE, ошибка из терминала, фрагмент документации из браузера. Всё, что можно выделить и скопировать через Ctrl+C, можно отправить агенту.

### Сбор контекста из нескольких файлов

Вам нужно показать агенту код из нескольких мест и дать одну команду. Вот как это делается:

1. Откройте первый файл, выделите нужный фрагмент. Нажмите F9 — код из буфера добавится в контекст. Появится попап «Контекст» с содержимым.

2. Откройте второй файл, выделите другой фрагмент. Нажмите F10, скажите «и вот этот интерфейс». Выделенный код + голосовой комментарий добавятся в контекст. Появится второй попап «Контекст».

3. Когда всё собрано — нажмите F7 и скажите «зарефактори оба компонента, выдели общий базовый класс». Все контексты + финальная команда уйдут агенту одним промптом. Контекстные попапы исчезнут.

### Управление сессиями

Вы закончили работу над одной задачей и хотите переключиться на другую с чистого листа. Нажмите F11 (SendTag с тегом `new-session`) — OpenVoicer создаст новую сессию, удалит старую. Контекст предыдущего диалога не будет мешать.

Если хотите начать новую сессию сразу с задачей — нажмите горячую клавишу с тегом `new-session` и одновременно надиктуйте команду. Новая сессия создастся, и ваша команда станет первым сообщением в ней.

### Отмена ошибочного запроса

Вы отправили команду, но передумали. На попапе «Обработка...» нажмите «Отменить». OpenCode прервёт генерацию, вы увидите попап «Отменено». Можно сразу отправлять новую, правильную команду — ничего не зависнет.

## Настройка подключения

### Базовая настройка: Voicer + OpenVoicer на одном компьютере

Это самый простой и типичный сценарий. Оба приложения работают локально, общаются через localhost.

В настройках Voicer убедитесь, что WS-порт установлен (по умолчанию 5050). В настройках OpenVoicer укажите тот же порт. При запуске OpenVoicer автоматически подключится к Voicer и отправит claim. Когда в трее OpenVoicer появится зелёная точка — можно работать.

### OpenCode через WSL2

Если вы разрабатываете в Linux-окружении через WSL2, OpenVoicer может запускать OpenCode внутри WSL. В настройках включите «WSL2», при необходимости укажите имя дистрибутива (если их несколько; пустое значение = default). В поле «Рабочий каталог» укажите путь к проекту внутри WSL — например, `~/projects/myapp` или `/home/user/code`. OpenVoicer запустит opencode serve через interactive bash, так что все переменные окружения из .bashrc будут доступны.

Порт OpenCode прокидывается через WSL2 автоматически — никаких дополнительных настроек проброса портов не требуется.

### OpenCode напрямую на Windows

Если opencode установлен нативно на Windows и доступен в PATH — снимите галочку «WSL2» в настройках OpenVoicer. Укажите рабочий каталог как Windows-путь (например, `D:\Projects\MyApp`). OpenVoicer запустит opencode serve напрямую.

### Выбор модели и агента

Откройте настройки OpenVoicer, перейдите в секцию «Модель». Нажмите «Обновить» — OpenVoicer загрузит список доступных провайдеров и моделей из API OpenCode. Выберите провайдера и модель из выпадающих списков. В секции «Агент» выберите агента — по умолчанию используется `build` (разработка). Изменения применяются немедленно.

## Архитектура проекта

Проект организован как монорепозиторий с чётким разделением ответственности.

```
Voicer/
├── src/
│   ├── Voicer.Core/              # Платформо-независимое ядро
│   │   ├── Models/               # AppSettings, HotkeyAction
│   │   ├── Services/             # SpeechRecognitionService, VoicerWebSocketServer
│   │   └── Interfaces/           # IAudioService, IHotkeyService, IPasteService
│   │
│   ├── Voicer.Desktop/           # Avalonia UI приложение
│   │   ├── App.axaml.cs          # Трей, меню, попапы
│   │   ├── AppOrchestrator.cs    # Стейт-машина: Idle→Recording→Processing
│   │   └── Views/                # SettingsWindow, TranscriptionPopup
│   │
│   ├── Voicer.Platform.Windows/  # NAudio, SetWindowsHookEx, SendInput
│   ├── Voicer.Platform.macOS/    # CoreAudio, CGEventTap
│   ├── Voicer.Platform.Linux/    # PulseAudio, X11 XGrabKey
│   │
│   └── OpenVoicer/               # Отдельное приложение-мост к OpenCode
│       ├── App.axaml.cs          # Трей, попапы, маршрутизация сообщений
│       ├── Services/
│       │   ├── VoicerWsClient.cs         # WS-клиент к Voicer
│       │   ├── OpenCodeClient.cs         # HTTP-клиент к OpenCode API
│       │   ├── OpenCodeEventService.cs   # SSE-клиент (события агента)
│       │   └── OpenCodeProcessManager.cs # Управление процессом OpenCode
│       └── Views/                # SettingsWindow, NotificationPopup
│
├── installer/                    # Инсталляторы для всех платформ
├── scripts/                      # Скрипты сборки и утилиты
└── docs/                         # Документация
```

**Voicer.Core** содержит всю логику, которая не зависит от операционной системы: настройки, распознавание речи, WebSocket-сервер. Платформенные различия абстрагированы через интерфейсы.

**Voicer.Platform.*** — реализации платформенных интерфейсов. Каждый проект использует нативные API своей ОС: NAudio и WASAPI для Windows, CoreAudio для macOS, PulseAudio для Linux. Горячие клавиши перехватываются через низкоуровневые хуки (SetWindowsHookEx, CGEventTap, X11 XGrabKey). Вставка текста эмулируется через нативные события ввода.

**Voicer.Desktop** — Avalonia UI приложение, которое склеивает всё вместе: DI-контейнер, трей-иконка, окно настроек, попапы. AppOrchestrator реализует стейт-машину с корректными переходами между состояниями.

**OpenVoicer** — отдельное приложение со своим DI, настройками и UI. Общается с Voicer по WebSocket, с OpenCode по HTTP и SSE. Управляет процессом OpenCode, накапливает контекст, показывает уведомления.

## Протоколы взаимодействия

### Voicer ↔ OpenVoicer: WebSocket

Подробная спецификация протокола: [websocket-protocol.md](websocket-protocol.md)

Ключевые моменты:
- Транспорт: `ws://localhost:{port}` (по умолчанию 5050)
- Claim-протокол: один активный клиент получает транскрипции
- Формат: JSON с полями `type`, `text`, `context`, `tag`, `timestamp`
- Автопереподключение с экспоненциальным backoff (от 1с до 10с)
- Таймаут чтения 60 секунд для обнаружения обрыва соединения

### OpenVoicer ↔ OpenCode: HTTP + SSE

Взаимодействие с OpenCode ведётся по двум каналам.

**HTTP API** используется для отправки команд и управления:
- `POST /session` — создать новую сессию
- `DELETE /session/{id}` — удалить сессию
- `POST /session/{id}/prompt_async` — отправить промпт агенту
- `POST /session/{id}/abort` — отменить текущий промпт
- `GET /provider` — получить список провайдеров и моделей
- `GET /agent` — получить список доступных агентов
- `PATCH /config` — изменить конфигурацию (модель, агент)

**SSE (Server-Sent Events)** используется для получения событий от агента в реальном времени. OpenVoicer поддерживает постоянное соединение к `GET /event` и обрабатывает следующие события:

| Событие | Действие OpenVoicer |
|---------|---------------------|
| `session.status` → `busy` | Показать попап «Обработка...», сбросить флаг отмены |
| `message.part.delta` | Накапливать токены ответа агента |
| `message.part.updated` | Обновить полный текст ответа |
| `message.updated` | Запомнить имя агента для отображения в попапе |
| `session.status` → `idle` | Показать попап «Готово» с ответом (или «Отменено» при отмене) |
