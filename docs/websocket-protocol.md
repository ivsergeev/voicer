# WebSocket Protocol

Voicer WebSocket-сервер слушает на `ws://0.0.0.0:{port}` (по умолчанию порт 5050, настраивается). К серверу может подключаться любое количество клиентов одновременно.

## Claim-протокол

Только один клиент одновременно является **активным** получателем транскрипций. Это предотвращает дублирование — например, когда несколько экземпляров клиента подключены к одному серверу.

- При подключении клиент **не** получает claim автоматически — он должен явно отправить `claim`
- Любой клиент может отправить `claim`, чтобы стать активным (последний claim выигрывает)
- При отключении активного клиента claim **сбрасывается** — автоматическая передача другому клиенту не происходит
- При `release` claim также сбрасывается без передачи другому клиенту
- Транскрипции отправляются **только** активному клиенту (если никто не взял claim — транскрипции не доставляются)
- Статусы и ошибки транслируются **всем** клиентам

## Сообщения: Клиент -> Сервер

### claim

Запрос на получение роли активного клиента. Предыдущий активный клиент получит уведомление `claimed` с `active: false`.

```json
{"type": "claim"}
```

### release

Добровольный отказ от роли активного клиента. Claim сбрасывается — автоматическая передача другому клиенту не происходит.

```json
{"type": "release"}
```

### ack

Обратная связь от клиента о статусе обработки транскрипции. Позволяет Voicer показывать пользователю, что происходит на стороне клиента. Voicer отображает popup-уведомления для `done` и `error`.

```json
{"type": "ack", "status": "submitted"}
{"type": "ack", "status": "responding"}
{"type": "ack", "status": "done"}
{"type": "ack", "status": "error", "message": "Описание ошибки"}
```

| status | Значение |
|--------|----------|
| `submitted` | Клиент успешно получил и отправил транскрипцию на обработку |
| `responding` | Клиент начал формировать ответ |
| `done` | Клиент закончил обработку |
| `error` | Ошибка при обработке транскрипции (подробности в `message`) |

## Сообщения: Сервер -> Клиент

### claimed

Подтверждение статуса claim. Отправляется при подключении, после `claim`/`release`, и при смене активного клиента.

```json
{"type": "claimed", "active": true}
{"type": "claimed", "active": false}
```

### transcription

Распознанный текст. **Отправляется только активному клиенту.**

```json
{"type": "transcription", "text": "Распознанный текст.", "timestamp": "2025-01-15T12:30:00.000Z"}
```

Если использовалась горячая клавиша «WS + selection» (F8 по умолчанию), выделенный текст (или содержимое буфера обмена, если ничего не выделено) добавляется в начало поля `text` через пробел:

```json
{"type": "transcription", "text": "выделенный текст Надиктованный текст.", "timestamp": "2025-01-15T12:30:00.000Z"}
```

### status

Изменение состояния микрофона. **Отправляется всем клиентам.**

```json
{"type": "status", "status": "recording"}
{"type": "status", "status": "processing"}
{"type": "status", "status": "idle"}
```

| status | Значение |
|--------|----------|
| `idle` | Ожидание, микрофон выключен |
| `recording` | Запись — пользователь удерживает горячую клавишу |
| `processing` | Распознавание речи в процессе |

### error

Ошибка распознавания или другая ошибка. **Отправляется всем клиентам.**

```json
{"type": "error", "message": "Описание ошибки"}
```

## Пример взаимодействия

```
Клиент A подключается       -> Сервер отправляет A: {"type":"claimed","active":false}
Клиент B подключается       -> Сервер отправляет B: {"type":"claimed","active":false}
Клиент A отправляет claim   -> Сервер отправляет A: {"type":"claimed","active":true}
Клиент B отправляет claim   -> Сервер отправляет A: {"type":"claimed","active":false}
                             -> Сервер отправляет B: {"type":"claimed","active":true}
Пользователь диктует        -> Сервер отправляет B: {"type":"transcription","text":"..."}
                             -> Сервер отправляет A и B: {"type":"status","status":"..."}
Клиент B отключается        -> Claim сбрасывается (не передаётся A)
Клиент A отправляет claim   -> Сервер отправляет A: {"type":"claimed","active":true}
Клиент A отправляет release  -> Сервер отправляет A: {"type":"claimed","active":false}
                             -> activeClient = null

Пользователь диктует (с ack):
Сервер -> B: {"type":"status","status":"recording"}
Сервер -> B: {"type":"status","status":"processing"}
Сервер -> B: {"type":"transcription","text":"Привет мир"}
B -> Сервер: {"type":"ack","status":"submitted"}
B -> Сервер: {"type":"ack","status":"responding"}
B -> Сервер: {"type":"ack","status":"done"}
```

## Подключение из WSL2

Из WSL2 Windows-хост доступен по IP из `/etc/resolv.conf`:

```bash
export VOICER_URL="ws://$(grep nameserver /etc/resolv.conf | awk '{print $2}'):5050"
```
